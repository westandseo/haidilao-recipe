# 하딜고고(Haidilgogo)

## 프로젝트 개요
하이디라오(훠궈) 소스/레시피를 공유하는 개인 프로젝트 웹사이트. 데스크톱/모바일 대응.
빌드 도구나 프레임워크 없이 순수 HTML/CSS/JS로 작성됨. 라이브 사이트: https://haidilgogo.github.io/

## 배포
- GitHub Pages로 배포 (조직 haidilgogo / 저장소 haidilgogo.github.io — 루트 주소로 서빙)
- 별도 빌드 과정 없음 — 파일 수정 후 push하면 그대로 반영됨
- 배포 브랜치: main

## 폴더/파일 구조
```
Haidilgogo/
├── index.html      # 페이지 구조 (탭, 검색창, 카드 그리드, 모달)
├── styles.css       # 전체 스타일
├── script.js        # 데이터 + 렌더링 로직 (아래 참고)
├── manifest.json    # PWA 매니페스트
└── assets/
    ├── cards/       # 레시피 카드 썸네일 (파일명 = 소스명.jpg, 버전 있으면 소스명_연도.jpg. 예: 건희소스[단]_2021.jpg, 영지소스_2025.jpg — 'recipe card_' 접두어 없음)
    ├── icons/       # 앱 아이콘·og (apple-touch-icon, icon-192, icon-512, logo-mark, og-image)
    └── fonts/       # 웹폰트 woff2
```
- **새 카드 이미지는 `assets/cards/`에** 넣고, `RECIPES`의 `img`는 `'assets/cards/<소스명>.jpg'`로 참조. 파일명에 대괄호([단]/[짠] 등) 있어도 브라우저에서 정상 로드됨(확인 완료).

## 데이터 구조 (script.js 안에 하드코딩, 별도 JSON 없음)
- `CATS`: 카테고리 목록 (`전체, 소스, 육수, 밥, 면, 기타`)
- `RECIPES`: 레시피 배열. 각 항목 필드:
  - `id`, `cat`(카테고리), `emoji`(썸네일 이미지 없는 카드의 배경 아이콘), `name`, `desc`(설명/유래)
  - `img`(이미지 경로, 없으면 emoji+tint 배경 사용), `imgBg`, `imgFit`(항상 `'cover'`), `tint`(그라데이션)
  - `order`: 주문 항목 (선택, 예: 밥 메뉴의 "공깃밥 1공기"). 이름은 `ORDER_ITEMS` 목록 안에서만 사용
  - `ings`: 재료 배열 `[이름, 양, 단위]`. 이름은 `SAUCE_BAR`, 단위는 `UNITS` 목록 안에서만 사용
  - `steps`: 조리 단계 배열 (없으면 `[]`)
  - `tip`: 팁 문구 (없으면 `''`)
  - `source`: 출처 표기 (선택, 예: "버블 건희(ONEUS)", "네이버블로그 sjsilver23"). `img`가 있는 카드에서만 썸네일 하단 그라데이션 위에 표시됨 — 출처가 불분명한 레시피는 생략 가능

## 썸네일 규격 (2026-07-11 규격화)
- 모든 카드 썸네일은 **1:1 정사각형**으로 통일(데스크탑·모바일 동일). `.recipe-thumb`가 `aspect-ratio: 1/1`, 이미지는 `object-fit: cover`로 꽉 채움.
- **카드 이미지는 반드시 정사각형 원본으로 준비할 것**(권장 1080×1080). 정사각형이 아닌 이미지는 `cover`로 가운데가 잘려 내용(특히 옆쪽 레시피 글씨)이 사라짐.
- 이미지 최적화: `sips -s format jpeg -s formatOptions 82 --resampleWidth 1080 --resampleHeight 1080 "원본" --out "결과.jpg"` (정사각형 원본 전제). 원본 PNG는 최적화 후 삭제.
- **개별 위치 조정(`imgPosition`)·카드별 scale 땜빵 금지.** 과거 영지소스 전용 scale 1.3 땜빵이 데스크탑 hover 규칙과 충돌해 "좁은 폭에서 hover 시 축소" 버그를 유발했음 → 규격화로 제거함. 크롭이 안 맞으면 코드로 때우지 말고 **정사각형 이미지 자체를 다시 준비**한다.
- hover/active 반응은 전 카드 공통(이미지 `scale(1.06)` + 카드 살짝 올라옴 + 테두리 브랜드빨강 1px). 카드별로 다르게 주지 않는다.
- **hover 확대 안전여백**: 이미지가 1.06배(가운데 기준) 확대되며 각 변 약 31px가 잘림. 핵심 내용(글씨·얼굴)은 1080 기준 **상하좌우 40px 여백 안(중앙 ~1000×1000)** 에 배치. 하단 ~15%는 어두운 그라데이션+출처 글씨 영역이라 핵심 배치 지양.

## 정식 재료·단위 목록 (표기 일관성용)
`script.js` 상단에 `SAUCE_BAR`(소스바 재료), `ORDER_ITEMS`(주문 항목), `UNITS`(단위)를 상수로 고정해둠.
페이지 로딩 시 모든 레시피를 검사해 **목록에 없는 이름/단위가 있으면 브라우저 콘솔에 경고**를 띄움 → 오타·표기 불일치(같은 재료 다른 이름) 방지.
- `매운소고기소스`와 `매운소고기소스[건더기만]`은 **의도적으로 구분**한 별개 항목 (통일하지 말 것)
- 진짜 새 재료라면 → 먼저 해당 목록에 추가한 뒤 레시피에 사용

### 재료 표시 순서 (자동 정렬, 2026-07-11)
`SAUCE_BAR` 배열의 **나열 순서 = 재료 서빙 순서**다. 레시피 상세를 렌더할 때 `sortIngs()`가 `ings`를 이 순서로 **자동 정렬**하므로, `RECIPES`의 `ings`는 아무 순서로 넣어도 화면엔 항상 같은 순서로 나온다(원본 데이터는 안 바뀜).
- 순서 그룹: **1) 베이스 소스 → 2) 액체 양념(굴소스·간장소스·중국식초) → 3) 고기·건더기 → 4) 채소·향신 → 5) 가루류 → 6) 기름류 → 7) 간·기타(설탕·소금·미원)**
- 그래서 **새 재료를 SAUCE_BAR에 추가할 땐 알맞은 그룹 위치에 넣을 것** (그 위치가 곧 표시 순서가 됨). 같은 그룹 안 순서는 SAUCE_BAR에 적힌 순서를 따름.
- 레시피 추가 시 `ings` 순서는 신경 쓸 필요 없음 — 자동 정렬됨.

### 레시피 추가 시 재료 검증 절차 (필수)
새 레시피 카드를 추가할 때는 **항상** `RECIPES`에 넣기 전에 모든 재료·단위·주문항목 이름을 정식 목록(`SAUCE_BAR` / `ORDER_ITEMS` / `UNITS`)과 대조 검증한다. 각 이름에 대해:

1. **정식 목록에 정확히 있으면** → 그대로 사용.
2. **목록에 없는데 기존 항목의 변형처럼 보이면** (띄어쓰기·동의어·줄임말·표기 차이 등, 예: `다진 파` vs `다진파`, `고춧가루` vs `마라시즈닝(고춧가루)`) → 임의로 합치거나 추가하지 말고 **사용자에게 먼저 확인**한다: "이거 기존 `○○`와 같은 건가요, 다른 건가요?"
   - 같음 → 기존 정식 이름으로 통일해서 사용.
   - 다름 → 3번으로.
3. **처음 보는(진짜 새) 재료면** → 정식 명칭을 사용자에게 확인한 뒤, 해당 목록(`SAUCE_BAR`/`ORDER_ITEMS`/`UNITS`)에 **먼저 추가**하고 나서 레시피에 사용한다.
4. **어떤 경우에도 말없이 새 이름을 만들거나 임의로 합치지 않는다.** 비슷해 보여도 의도적으로 구분된 항목이 있음 (예: `매운소고기소스` vs `매운소고기소스[건더기만]`). 애매하면 반드시 사용자에게 물어본다.

**표기 규칙: [대괄호]=변형·옵션 (예: `건희소스[단]`, `매운소고기소스[건더기만]`) / (소괄호)=정식명이 낯설 때 붙이는 고정 부연 (예: `마라시즈닝(고춧가루)`, `오향우육(다진고기)`).** 소괄호 부연은 재료마다 한 번 정하는 이름의 일부이며, 원작자가 레시피별로 다르게 부른 이름은 병기하지 않는다(카드 이미지에 원본이 있음).

즉 기본 원칙: **애매하거나 목록에 없으면 → 추측하지 말고 사용자에게 확인.**

### 단위(양) 표기 방침 (2026-07-11 확정: 원본 충실)
하이디라오 소스바 도구는 **지점마다 다르므로**(스푼/국자 등) "진짜 정답 도구"는 없음 → **원작자(썸네일 원본) 표기를 그대로 따르는 것**이 유일하게 검증 가능한 기준이다.
- **레시피마다 단위가 달라도 정상**이다(사용자는 한 번에 소스 하나만 만들며, 그 레시피 안 비율만 맞으면 됨). 재료별로 단위를 강제 통일하지 말 것 — 원본을 훼손함.
- 허용되는 정규화는 **동의어 도구명 통일(국자·숟갈 → 스푼)** 과 **애매어 정규화(많이→넉넉하게, 조금→적당히, 반→0.5)** 뿐. `티스푼`은 스푼과 다른 크기이므로 원본대로 유지. `바퀴`(두르기)도 원본이 그리 적었으면 유지.
- **스푼을 집게로 바꾸는 식의 도구 재해석 금지**(뜻이 달라짐). 과거 다진파를 임의로 `집게`로 적었던 것을 2026-07-11 전부 원본 `스푼`으로 되돌림(양은 원본과 동일했음).

## 주요 기능 (script.js)
- 카테고리 탭 필터링 + 검색 (이름/재료명 매칭, 설명은 검색 대상 아님 — 설명 문구의 단어로 걸려서 원치 않는 레시피가 나오는 걸 막기 위해 의도적으로 제외)
- 카드 클릭 → 상세(주문/재료/단계/팁) 표시. 데스크톱=가운데 모달 / 모바일(≤640px)=상단바 아래 전체화면 패널(상단바는 그대로 동작하며, 탭·검색 누르면 상세가 자동으로 닫히고 그 동작 실행. X버튼으로 닫음, history 처리 없음)
- 즐겨찾기는 `localStorage`에 저장되어 새로고침/재방문 후에도 유지됨
- 커스텀 pull-to-refresh는 제거됨(e5807a6). 로고 클릭 시 `?_r=timestamp` 쿼리로 강제 새로고침

## 작업 시 유의사항
- 사용자는 코딩 초보 — 변경사항은 단계별로, 스크린샷/결과 확인 요청 가능성 있음
- 디자인 변경 시 기존 tint/이모지 톤과 일관성 유지
- 프레임워크 도입 등 구조를 바꾸는 제안은 먼저 설명 후 진행
